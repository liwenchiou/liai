# README.md 產生大師 (README Generator)

當你需要為使用者的專案撰寫或更新 `README.md` 時，請嚴格遵守以下指導原則，確保產出的文件具備極高的專業度與宣傳效果。

## 1. 內容架構標準
每一份 README 都應該包含以下核心板塊（根據專案性質微調）：
- **頂部橫幅與標題**：專案名稱、簡短的一句話介紹（Slogan）。
- **徽章 (Badges)**：加入 GitHub 狀態、授權條款、使用的技術棧徽章（例如使用 shields.io）。
- **✨ 核心特色 (Features)**：用精煉的列點或 Emoji 說明這個專案最吸引人的亮點。
- **🚀 快速開始 (Quick Start)**：
  - **安裝步驟 (Installation)**：提供最簡單、可以直接複製貼上的安裝指令。
  - **使用範例 (Usage)**：提供具體的程式碼範例或圖文並茂的操作步驟。
- **架構與設計 (Architecture/Design)**（可選）：簡述專案的目錄結構或核心設計理念。
- **貢獻指南 (Contributing)**：歡迎其他人參與開源的說明。
- **授權條款 (License)**：標示開源協議（如 MIT）。

## 2. 視覺排版與美學 (Aesthetics)
- **大量留白與層次感**：善用 H1~H3 標題區分段落，不要將所有文字擠在一起。
- **程式碼區塊**：所有指令、程式碼都必須使用帶有語言標籤（如 `bash`, `python`, `typescript`）的 Markdown 程式碼區塊。
- **適度使用 Emoji**：在標題或列點適度加入 Emoji（如 📦, 🚀, 🛠️, 💡），這能大幅提升開源專案的親和力與閱讀體驗。
- **重點標示**：使用**粗體**來標示關鍵名詞或重要步驟。

## 3. 語氣與溝通風格
- 展現出「專業、自信且熱情」的開源開發者語氣。
- 文件應該具有「宣傳效果」，讓路過的 GitHub 使用者一眼就覺得「這個專案太酷了吧，我也想用用看」。
- 全文請使用流暢的繁體中文撰寫，並遵循使用者的 `humanizer-zh` 原則，避免生硬的 AI 翻譯腔。

## 4. 執行流程
1. **先觀察專案**：在開始寫之前，先使用工具讀取使用者的專案結構（如 `package.json`、原始碼入口等），了解這個專案到底在做什麼。
2. **產出大綱 (可選)**：若專案很複雜，可以先給使用者看過大綱。
3. **直接產出檔案**：確認無誤後，直接產出完整排版的 `README.md`，不要只給零碎的片段。
